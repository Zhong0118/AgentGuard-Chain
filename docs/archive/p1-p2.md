**关于 CLI 和 Web 怎么设计**
我的判断：**不要现在做完整 Web 控制台，也不要把 CoreCoder 的 CLI 重写成 Web。**

CoreCoder 已经有 CLI。如果我们再做一个完整 Web Agent UI，会变成两个 Agent 入口，工程会散。

更合理的是：

```text
CLI 负责运行 Agent
Dashboard/Web 负责观察审计日志
必要时 Web 只做 pending approval 控制台
```

也就是说：

```text
Agent 在 CLI 运行
Guard 嵌在工具执行链路里
审计日志写 JSONL
Dashboard 读 JSONL 展示风险
```

这是当前最稳的结构。

**交互确认怎么做最合理**
分阶段做，不要一步到 Web。

第一阶段，CLI 确认：

```text
CoreCoder CLI / MiniAgent runner
↓
AgentGuard 返回 ask
↓
CLI 打印风险说明
↓
用户输入 y/n
↓
执行或阻断
↓
写审计日志
```

这是最直接的，因为工具执行在 CLI 进程里，确认也在同一个进程里，不需要跨进程通信。

第二阶段，Web 观察：

```text
CLI 跑 Agent
↓
AgentGuard 写日志
↓
Dashboard 展示 allow/deny/ask/chain_alert
```

这只是观察，不控制执行。

第三阶段，Web 审批，后续再做：

```text
CLI Agent 进入 pending
↓
写 pending_actions.jsonl 或 pending_actions.json
↓
Dashboard 展示 pending action
↓
用户点击 approve/deny
↓
CLI 轮询 approval 文件
↓
继续执行或阻断
```

这个能做，但它涉及状态同步、轮询、超时、并发、失败恢复。现在做会把 P1 拉复杂。

所以推荐当前不做 Web 审批，先做 CLI ask。

**是不是开两个 CLI？**
不建议。

可以一个 CLI：

```text
CoreCoder guarded runner / MiniAgent runner
```

里面直接 ask：

```text
AgentGuard asks:
Tool: bash
Command: python setup.py install
Risk: command_not_allowed
Approve? [y/N]
```

Dashboard 另开一个浏览器页面只观察日志。

所以实际是：

```text
一个 CLI 运行 Agent + Guard
一个 Web Dashboard 看审计
```

不是两个 CLI，也不是 Web 负责判断。

**Web 输入内容、显示 log、让用户决定要不要做吗？**
这是后续高阶形态，可以做成：

```text
Web 输入任务
↓
后端启动 Agent
↓
Web 展示日志
↓
Web approve/deny
```

但这已经是一个完整 Web Agent 产品了，不是比赛 MVP 必须要先做的。

目前更稳的是：

```text
CLI Agent
+ Guard hook
+ JSONL audit
+ Dashboard display
```

**下一步我建议做什么**
按第一性原则，下一步应该补“工具结果审查 + 输出脱敏 + ask 三态闭环”的最小版本。

但顺序我建议是：

1. **先做工具结果审查**
因为这直接补齐 workflow 里的“调用结果内容审查”。

```text
工具执行后
↓
ResultInspector 检查输出
↓
发现 secret/token/password/private_key
↓
脱敏 result_preview
↓
必要时追加 risk_type
↓
审计日志记录 output_findings
```

2. **再做 ask 用户确认**
因为 ask 需要区分哪些风险不是直接 deny，比如：

```text
写入普通项目文件
删除临时文件
运行未在 allowlist 但低风险命令
访问外部网络但用户任务允许确认
```

3. **再做风险链结构化**
现在 `chain_alerts` 只是告警。后续应该变成：

```json
{
  "chain_id": "...",
  "chain_type": "SensitiveReadToExternalSend",
  "nodes": [...],
  "edges": [...],
  "severity": "critical"
}
```

4. **最后考虑 Web approval**
等 ask 和 pending action 稳了，再把 Dashboard 从观察升级为审批控制台。
